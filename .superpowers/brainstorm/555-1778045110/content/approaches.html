3 approaches for the navbar transformation

Same end result — different implementation strategies. My recommendation is highlighted.

A ★

Two separate components + scroll state (Recommended)

Keep the top navbar and the left pill as two separate JSX elements. A useScrolled hook flips a boolean at ~80px. Top navbar gets AnimatePresence exit animation (fade + slide up-left). Left pill has entrance animation (slide in from x:-60, y:-40 → 0,0) — giving the diagonal arrival feel. Both use motion.div.

Pros

  • Cleanest code — each state is its own component
  • Framer Motion handles all interpolation
  • Easy to tweak each state independently
  • Consistent with your existing codebase

Cons

  • Two components instead of one
B

Single component with Framer Motion layout animation

One <nav> component that uses Framer Motion's layout prop. On scroll, change its CSS from top-0 horizontal flex to left-0 vertical flex and let Framer Motion's FLIP algorithm animate the position morph automatically.

Pros

  • Elegant — one element, one truth
  • FLIP animation is buttery smooth

Cons

  • Layout animations on position:fixed elements are finicky
  • Hard to independently control each nav item's morph
  • Risk of jank at the transform boundary
C

Pure CSS transitions (no Framer Motion for the swap)

Toggle a CSS class on scroll. Two absolutely-positioned navs, each with transition: all 0.4s. No JS animation library for the swap — just opacity, transform, and pointer-events.

Pros

  • Lightest — no animation library overhead for this part

Cons

  • Less expressive — no spring physics, no staggered entry
  • Inconsistent with existing Framer Motion usage
  • Diagonal morph is harder to orchestrate in CSS

Option A is recommended — fits your stack, gives the most control over the diagonal animation. Confirm in terminal to see the full design.
